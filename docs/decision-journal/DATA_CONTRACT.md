# DECISION_JOURNAL_DATA_CONTRACT.md
# عقد بيانات سجل قرارات التحليل

---

## الهدف من العقد

يُعرَّف هذا الملف كـ **عقد بيانات (Data Contract)** لتوحيد شكل قرارات التحليل عبر المنصات والمراحل المختلفة للمشروع.

الهدف هو:
- تحديد هيكل موحد وثابت لقرارات اللجان قبل ربطها بقاعدة البيانات.
- ضمان الفصل الكامل بين منصتي MT5 و OKX.
- توفير مرجع واضح للمطور والمراجع لفهم شكل البيانات قبل البدء بالتطوير الفعلي.

---

## لماذا لا نربطه بقاعدة البيانات الآن؟

في المرحلة الحالية (A6)، الأولوية هي **تعريف الشكل لا التطبيق**. ربط العقد بـ Convex يتطلب:

1. تصميم Convex Tables تتطابق مع هذه الأنواع.
2. كتابة validators بشكل صحيح.
3. التأكد من سياسات الأمان (Security Rules) قبل كتابة أي بيانات.

هذه الخطوات ستُنفَّذ في مرحلة لاحقة بعد التحقق من استقرار العقد.

---

## الفرق بين `status` و `finalDecision`

| الحقل | المعنى | من يحدده |
|---|---|---|
| `status` | الحالة التشغيلية الحالية للإشارة | النظام تلقائياً |
| `finalDecision` | القرار النهائي الصادر عن مجموع اللجان | نتيجة تصويت اللجان |

### أمثلة توضيحية:

- إشارة بـ `status: "READY_FOR_REVIEW"` لكن `finalDecision: "BLOCK"` → الإشارة جاهزة للمراجعة لكن اللجان منعتها.
- إشارة بـ `status: "WATCHING"` لكن `finalDecision: "HOLD"` → النظام يراقب ولم يصدر قرار دخول بعد.
- إشارة بـ `status: "EXPIRED"` → انتهت صلاحيتها بصرف النظر عن القرار.

---

## الفرق بين منصة MT5 و منصة OKX

| البُعد | MT5 | OKX |
|---|---|---|
| **الوضع الحالي** | متصل — Read-only Bridge | Placeholder — غير مفعّل |
| **مسار الكود** | `src/lib/mt5`, `src/components/lab/mt5` | `src/lib/okx`, `src/components/lab/okx` |
| **API الحقيقي** | جسر Python Read-only | غير موجود حالياً |
| **التنفيذ** | ممنوع في الوضع الحالي | ممنوع كلياً |
| **الخلط** | ممنوع — كل منصة بياناتها منفصلة | ممنوع — كل منصة بياناتها منفصلة |

> ⚠️ **قاعدة صارمة:** أي `DecisionJournalEntry` يجب أن يحمل `platform` واحدة فقط. لا يوجد إدخال يجمع بيانات MT5 و OKX.

## كيف سيُستخدم مع Convex لاحقاً

في مرحلة مستقبلية سيتم:

1. إنشاء Convex Table باسم `decisionJournal` تتطابق حقولها مع `DecisionJournalEntry`.
2. كتابة `validator` في Convex يُطابق الأنواع المعرَّفة هنا.
3. استخدام `useQuery(api.decisionJournal.list)` في صفحة Decision Journal.
4. كتابة `mutation` خاص بالإضافة فقط — لا تعديل ولا حذف.
5. تطبيق Row-Level Security لضمان أن كل مستخدم يرى قراراته فقط.

---

## كيف سيُستخدم مع صفحة Decision Journal لاحقاً

الصفحة الحالية (`/decision-journal`) تعرض Empty State و جدول بأعمدة ثابتة.

في المرحلة القادمة:
1. تستورد `DecisionJournalEntry` من هذا الملف.
2. تستخدم Convex query لجلب السجلات.
3. تعرض كل حقل من `DecisionJournalEntry` في العمود المناسب.
4. تُلوّن الحالات (status, grade, verdict) بواسطة مكونات Badge موجودة.

## القواعد الأمنية الإلزامية

| القاعدة | التفصيل |
|---|---|
| 🚫 لا تنفيذ من السجل | يُحظر وجود زر "تنفيذ" أو "فتح صفقة" في صفحة Decision Journal |
| 🚫 لا حذف قرارات | القرارات المسجلة لا تُحذف — Immutable Log |
| 🚫 لا تعديل قرارات | `readOnly: true` إلزامي في كل إدخال |
| 🚫 لا خلط MT5 و OKX | كل إدخال يحمل `platform` واحدة — لا دمج |
| ⚠️ OKX Read-only حالياً | حتى عند تفعيل OKX، ستبقى واجهة القرارات للعرض فقط |
