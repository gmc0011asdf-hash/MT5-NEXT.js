# تقرير الفحص الهيكلي للمشروع (PROJECT STRUCTURE AUDIT)
**تاريخ الفحص:** 2026-05-02
**الأنظمة المستهدفة:** نظام الملك الهندسي للتداول العالمي (MT5 / OKX)

## 1. ملخص عام عن المشروع الحالي
المشروع عبارة عن منصة تداول وتحليل مؤسسية حديثة مبنية باستخدام:
- **الواجهة الأمامية (Frontend):** Next.js (App Router)، React 19، Tailwind CSS، مع مكتبات واجهة مثل shadcn/ui و lucide-react. الواجهة تدعم اللغة العربية (RTL).
- **الواجهة الخلفية (Backend/Database):** Convex (Real-time Database + Serverless Functions).
- **المصادقة (Authentication):** Clerk.
- **الربط مع منصات التداول (Bridge):** خدمة وسيطة مبنية بـ Python (FastAPI) للقراءة فقط من MetaTrader 5 (MT5 Readonly Service).

النظام حالياً مصمم للتكامل بشكل عميق مع MT5، ويحتوي على بيئة مختبر (Lab) لتحليل الإشارات، وتقارير (Reports)، وإعدادات (Settings).

## 2. هيكل المجلدات الرئيسي
- `src/app`: يحتوي على مسارات Next.js (الداشبورد، تسجيل الدخول، إلخ).
- `src/components`: مكونات الواجهة مقسمة وظيفياً (dashboard, lab, reports, settings, layout, ui).
- `src/lib`: دوال مساعدة، هوكات، وثوابت (Hooks, Constants, utils).
- `convex/`: ملفات الواجهة الخلفية، المخططات (Schema)، والاستعلامات.
- `mt5_readonly_service/`: خدمة بايثون للاتصال بـ MT5 وقراءة البيانات.

## 3. ملفات الواجهة المهمة
- `src/app/layout.tsx`: المخطط الجذري للتطبيق.
- `src/app/(dashboard)/layout.tsx`: مخطط الداشبورد الجانبي والعلوي الذي تم تعديله ليتوافق مع RTL.
- `src/app/(dashboard)/dashboard/page.tsx`: الصفحة الرئيسية لعرض النظرة العامة.

## 4. ملفات المختبر (Lab) الحالية
- `src/app/(dashboard)/lab/page.tsx`: الواجهة الأساسية لمختبر التداول.
- `src/components/lab/`: يحتوي على المكونات الفرعية الخاصة بتحليل الإشارات، عرض اللجان، وتقييم الصفقات.

## 5. ملفات MT5 الحالية
- `mt5_readonly_service/main.py`: خدمة بايثون للاتصال بـ MT5 وسحب البيانات (محددة للقراءة فقط).
- `convex/mt5Bridge.ts`: واجهة Convex البرمجية لاستقبال ومعالجة بيانات MT5 وتخزينها في قاعدة البيانات.
- `src/lib/mt5-bridge/`: مكتبات ومساعدات الواجهة الأمامية للتعامل مع بيانات MT5.
- جداول `schema.ts` مثل `mt5AccountSnapshots` و `mt5MarketTicks` و `mt5OpenPositions`.

## 6. ملفات OKX
لا توجد حالياً أي ملفات أو مجلدات أو إشارات برمجية تخص OKX في المشروع. النظام بأكمله مخصص لـ MT5 حالياً.

## 7. ملفات Convex / Backend / Database
- `convex/schema.ts`: يحدد كافة جداول قاعدة البيانات بما فيها الحسابات، الصفقات، المختبر، التقارير.
- `convex/coreQueries.ts`: يحتوي على الاستعلامات الأساسية لعرض البيانات في الواجهة.
- `convex/technicalIndicators.ts`: منطق حساب المؤشرات الفنية.

## 8. ملفات Clerk / Auth
- `src/app/sign-in` و `src/app/sign-up`: صفحات تسجيل الدخول والاشتراك.
- المكونات المغلفة لـ Clerk (Providers) في `src/components/providers/`.
- إعدادات Clerk في الجذور ومزامنة المستخدمين مع Convex `users`.

## 9. ملفات التقارير (Reports)
- `src/app/(dashboard)/reports/page.tsx`: واجهة التقارير الشاملة.
- `src/components/reports/`: المكونات الخاصة برسم المخططات وعرض سجل الصفقات والمقاييس.

## 10. ملفات الإعدادات (Settings)
- `src/app/(dashboard)/settings/page.tsx`: واجهة الإعدادات.
- `src/components/settings/`: مكونات ضبط إعدادات الحساب، التحكم في MT5، وضوابط المخاطر.

## 11. ملفات التنفيذ أو التداول (التي يجب الحذر منها)
- لا يوجد كود تنفيذ صفقات حي في `mt5_readonly_service`. الخدمة هي `readonly` فعلياً.
- `convex/schema.ts`: يحتوي على جدول `governanceState` الذي يدير `tradingEnabled`.
- أزرار مختبر التداول (Lab) قد تحتوي مستقبلاً على أزرار "تنفيذ"، لكنها حالياً معزولة عن أي مسار تنفيذ حقيقي.

## 12. الملفات المكررة أو القديمة (Legacy)
- `src/app/(dashboard)/layout.tsx.bak`: نسخة احتياطية قديمة من مخطط الداشبورد يجب إزالتها لتجنب التشتت.
- العديد من ملفات التخطيط (`*_REVIEW.md`, `*_PLAN.md`) في الجذر والتي قد تكون انتهت صلاحيتها وتحتاج إلى الأرشفة.

## 13. الملفات غير المستخدمة المحتملة
- `convex/testEvents.ts` و `convex/coreSeed.ts`: ملفات مخصصة للاختبار والبذر (Seeding) والتي قد لا تكون مطلوبة في بيئة الإنتاج الصارمة.

## 14. المشاكل المعمارية الحالية
- **الارتباط الوثيق (Tight Coupling):** جداول قاعدة البيانات وواجهات الاستعلام تحمل بادئة `mt5` (مثل `mt5OpenPositions`). إذا تم دمج OKX مباشرة، سيتم تكرار الجداول والمكونات، مما يخلق بيئة غير قابلة للتوسع والصيانة.
- **غياب التجريد (Lack of Abstraction):** لا توجد طبقة وسيطة (Interface/Adapter) في الواجهة الأمامية لتوحيد بيانات MT5 و OKX لتُعرض في مكون واحد كـ `PositionTable`.

## 15. أين يجب فصل MT5
- **في قاعدة البيانات:** يجب فصل الجداول إما عبر إنشاء جداول موحدة مع حقل `provider: "MT5" | "OKX"`، أو الإبقاء عليها مفصولة مع إنشاء استعلامات (Queries) موحدة تجمع بينهما في الخلفية.
- **في خدمات الواجهة الخلفية:** الاحتفاظ بـ `mt5Bridge.ts` كمسار خاص بـ MT5 فقط دون دمج أي منطق خاص بـ OKX داخله.
- **في الواجهة الأمامية:** يجب أن تعتمد مكونات الـ UI على واجهات بيانات مجردة (Abstract Data Types) بدلاً من استهلاك كائنات MT5 بشكل مباشر.

## 16. أين يجب إضافة OKX لاحقًا
- إنشاء خدمة قراءة منفصلة `okx_readonly_service` (سواء بـ Python أو Node.js).
- إنشاء `convex/okxBridge.ts` منفصل كلياً لمعالجة بيانات OKX وحفظها.
- إنشاء `src/lib/okx-bridge/` في الواجهة الأمامية للتعامل مع منطق الاتصال الخاص بـ OKX.
- تضمين مفتاح التبديل (Feature Toggle / Provider Switch) في الواجهة (Shared UI) لتحديد سياق عرض البيانات (MT5 أو OKX).

## 17. المخاطر التي يجب عدم لمسها الآن
- دوال وخدمات بايثون الخاصة بالاتصال بـ MT5 يجب أن تظل Read-only لتفادي أي كارثة تنفيذية غير مقصودة.
- هيكل `schema.ts` المرتبط بصلاحيات المستخدم والمصادقة.
- استقرار تصميم واجهة المستخدم العربية (RTL Sidebar Layout) الذي تم ضبطه مؤخراً.

## 18. توصية المرحلة التالية (Stage 2)
1. **أرشفة وتنظيف:** إزالة الملفات الاحتياطية (`.bak`) وتنظيم ملفات الـ `MD` في مجلد للتوثيق (`docs/`).
2. **إعادة هيكلة قواعد البيانات والاستعلامات (Abstraction):** تصميم الواجهات البرمجية (Interfaces) و Data Access Layer في `convex` ليكون `Provider-Agnostic` لتسهيل إضافة OKX مستقبلاً.
3. **تأسيس معمارية OKX المعزولة:** تهيئة المجلدات الأساسية لخدمات OKX (دون تفعيل التداول أو الرافعة المالية) بحيث تتكامل بيانات السوق والصفقات المفتوحة مع الواجهة الموحدة المستحدثة في النقطة السابقة.
